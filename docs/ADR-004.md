# ADR-004 — Observability + Evaluation Backbone

| Field          | Value                                                                |
| -------------- | -------------------------------------------------------------------- |
| **Status**     | Accepted                                                             |
| **Date**       | 2026-05-18                                                           |
| **Deciders**   | Putsch Platform Engineering, Datenschutzbeauftragte, Betriebsrat-IT  |
| **Supersedes** | —                                                                    |

## Context

Putsch Group is rolling out a production agentic stack — CrewAI +
LangGraph orchestration over Mistral La Plateforme, with Docling for
document ingest and Zep+Graphiti for temporal memory. The stack is
deployed in the Putsch Frankfurt VPC.

We need an observability + evaluation layer that:

1. **Stays in the EU sovereign perimeter.** All traces, prompts, model
   outputs, and eval data must be hosted on infrastructure under Putsch
   control. Vendor-managed offerings are disqualified by the
   Datenschutz-Folgenabschätzung.
2. **Closes the production-to-training loop.** Every production trace
   must be a candidate eval item. Sachbearbeiter annotations must feed
   directly into the DSPy GEPA training set without manual ETL.
3. **Treats PII redaction as a cryptographic concern.** Redaction must
   happen *before* data leaves the application boundary, must be
   reversible only via an audited vault, and must fail-closed.
4. **Is framework-agnostic.** Today: CrewAI + LangGraph. Tomorrow:
   whatever wins. The wire protocol must be OpenTelemetry so the
   backend is a config change.

## Decision

We will build `putsch-obs`:

- **Self-hosted Langfuse** (MIT, OSS image, Postgres + ClickHouse +
  MinIO) deployed via `docker compose` on a Hetzner Frankfurt CCX33 pair
  behind a Hetzner LB.
- **OpenTelemetry as the wire protocol.** The application emits OTLP
  to a local OTel collector. The collector exports to Langfuse. Swapping
  Langfuse for Tempo/Jaeger/etc. is a collector config change.
- **PII redaction at the SDK level**, two-stage (deterministic regex →
  local Qwen3-14B LLM), fail-closed. Reversible via a Postgres-backed
  vault with an append-only, hash-chained audit log enforced by a WORM
  Postgres trigger.
- **Eval datasets live in git** at `evals/datasets/*.jsonl`. Synced into
  Langfuse Datasets on PR merge. The git artefact is canonical; UI-only
  edits are forbidden (CI fails if Langfuse contains items not in git).
- **Annotations close the loop**: Langfuse annotation queue items are
  promoted to DSPy training rows by `human_review.annotations_to_training_set`.

| LangSmith      | Proprietary        | Enterprise only  | closed-source, enterprise self-host gates features behind a contract               |
| Arize Phoenix  | Elastic License 2.0 | Yes              | Elastic License explicitly blocks managed offerings; OK for internal use but no    |
|                |                    |                  | future-proof on shifting around the licence                                        |
| Helicone       | MIT                | Proxy-only       | proxy intercept misses non-LLM spans (agents, tools, memory)                       |
| Opik (Comet)   | Apache 2.0         | Yes              | younger; framework coverage thinner; not chosen but documented as a backup         |
| Custom Tempo   | Apache 2.0         | Yes              | building eval + annotation + dataset UX from scratch is a 6-month project          |

### Why **not** layer Laminar alongside

Laminar's browser-agent trace UX is genuinely better for 2,000-span agent
runs. We chose to ship Langfuse first and add Laminar later if
Mahnverfahren swarm debugging becomes a top-3 toil source. OTel as the
wire protocol means adding Laminar is a collector config change, not a
rip-and-replace.

### Why PII redaction at the SDK and not at the Langfuse UI

The Langfuse UI redaction layer redacts *display*. The data still sits
in ClickHouse unredacted. That is the wrong threat model for Putsch:

- A Langfuse server compromise must not yield un-redacted PII.
- A Datenschutz-Folgenabschätzung audit must show that data is redacted
  *before* it traverses storage, not before it traverses the screen.
- The reversible-tokenization vault gives us a clean audit story
  (un-redaction is an action with a who, why, when).

### Why reversible tokenization and not irreversible hashing

Irreversible hashing makes investigations impossible. A
Sachbearbeiter needing to confirm "did the system route invoice X to
the right approver?" must be able to resolve a tokenized customer name
back. Hashing would force a "screenshot before redaction" practice that
defeats the whole point. The audited vault is the right shape: any
un-redaction is logged.

### Why the audit log is WORM-enforced at the database level

A process-level append-only log is fine until the process is
compromised. The WORM trigger in `001_init.sql` rejects UPDATE/DELETE at
the database engine, which means a Langfuse compromise — or a vault
process compromise short of `DROP TABLE` privileges — cannot rewrite
audit history. The hash chain detects even that case.

## Consequences

### Positive

- Full feature parity with Langfuse Cloud, in our own perimeter.
- Single backend for spans, evals, prompts, datasets, annotations.
- Switching to another OTel-native backend later is a config change.
- The DSGVO Verzeichnis von Verarbeitungstätigkeiten is auto-generated;
  the Datenschutzbeauftragte doesn't chase service teams.

### Negative

- We own the operational complexity: Postgres, ClickHouse, MinIO, Redis,
  Langfuse worker, Langfuse web, OTel collector. The runbook
  (`docs/runbook.md`) documents the 5 most likely failure modes.
- The PII redactor depends on a local Qwen3-14B endpoint. That endpoint
  is a new component to operate. Trade-off: redaction LLM-stage is
  fail-closed; if Qwen3 is down, traces drop rather than leak. We
  accept the lost-observability cost.
- Per-trace-type ClickHouse TTL requires careful schema migration when
  retention classes change. The runbook covers this.

### Neutral

- Langfuse v2 → v3 will require a migration. We've pinned the image
  version and noted the upgrade as a runbook activity. We will not
  auto-update.

## Compliance

- **DSGVO Art. 30**: auto-generated Verzeichnis in
  `src/putsch_obs/dsgvo/`. Each service must register itself.
- **DSGVO Art. 32**: redaction-at-source + WORM audit log fulfils the
  "appropriate technical measures" obligation. The audit trail
  dashboard is the auditable artefact.
- **EU AI Act Art. 12**: retention is per-trace-type, enforced at the
  ClickHouse TTL layer. Default: 10 years for HR/payroll, 3 years for
  customer-facing, 90 days for development.
- **§87 BetrVG**: annotation queues route to roles, not individuals.
  Per-individual scoring is suppressed in the dashboards. The Putsch
  Betriebsvereinbarung KI-Einsatz codifies this; the registration model
  carries a `LegalBasis.BETRVG_KI_EINSATZ` enum value for crews that
  fall under it.
